## 核心问题

模型何时应回答，何时应先调用工具。

## 这节学什么

学习“回答”和“行动”是两种不同能力，工具调用用于补足模型的外部能力。

工具调用是让模型把任务交给外部能力完成，例如搜索、计算、查库、执行脚本。ReAct 则是把这个过程组织成“思考 -> 行动 -> 观察 -> 再行动 -> 最终回答”的工作流。前者是能力接口，后者是推理框架；结合起来，模型才不会只靠参数记忆硬答。

## 关键概念

- tool use
- act-observe loop
- ReAct
- routing
- observation

## 一句话判断

可以直接回答的任务，通常是解释、翻译、改写、总结、创意生成、稳定常识。

应该先调用工具的任务，通常涉及实时信息、精确计算、私有数据、外部状态变更、程序执行或高风险核验。

## 典型练习

定义搜索和计算器两个工具，让模型完成先查再算的任务。

## 最小示例

模型先调用 `search` 获取汇率，再调用 `calculator` 完成换算，最后整理成答案。

一个更完整的最小循环可以写成：

1. 思考：我不知道实时汇率，不能直接答。
2. 行动：调用 `search("USD CNY exchange rate today")`
3. 观察：返回当前汇率，例如 `1 USD = 7.21 CNY`
4. 再行动：调用 `calculator("100 * 7.21")`
5. 观察：返回 `721`
6. 最终回答：今天按检索到的汇率，100 美元约等于 721 元人民币。

这里最重要的不是“会不会算”，而是模型先判断出“这个信息不该靠猜”。

## 何时直接回答，何时调用工具

可以把它理解成一个简单路由器：

1. 如果问题涉及“今天、刚刚、最新、当前、实时”，优先调用工具。
2. 如果问题要求高精度数值结果，优先调用计算或代码工具。
3. 如果答案依赖企业知识库、本地文件、数据库、用户资料，优先调用检索工具。
4. 如果任务会改变外部状态，例如发邮件、改日程、下单、删数据，必须调用工具，而且通常要有人类确认。
5. 如果任务只是概念解释、文本改写、写作、翻译、头脑风暴，通常可以直接回答。
6. 如果任务看似简单，但模型在字符计数、复杂表格、长列表筛选上容易出错，也应考虑调用工具。

## 课堂流程

1. 解释为什么模型不应凭空猜外部事实
2. 演示简单工具协议
3. 分析一次 act-observe 循环
4. 对比“直接回答”与“先调工具”的差异
5. 讨论失败案例与权限边界

## 五类例子

### 1. 搜索类

用户任务：总结昨晚苹果发布会的核心硬件参数。

为什么不能直接答：这是高度时效性信息，模型参数里不一定有。

应该调用的工具：`web_search`

观察：拿到芯片、屏幕、续航、价格等网页片段。

最后如何回答：按“新品名称 -> 关键参数 -> 与上一代差异”整理，而不是把搜索片段原样贴给用户。

### 2. 计算类

用户任务：本金 50000 元，年化 4.5%，复利 7 年，本息和是多少。

为什么不能直接答：复利计算容易出现数值幻觉。

应该调用的工具：`calculator` 或 `python`

观察：返回精确结果。

最后如何回答：给出公式、结果和一句人话解释，例如“若按年复利计算，7 年后约为 68043.16 元”。

### 3. 知识库检索类

用户任务：客户咨询某款加湿器坏了，能否退换。

为什么不能直接答：退换货政策属于企业私有规则，不应凭模型印象回答。

应该调用的工具：`query_knowledge_base`

观察：返回“7 天无理由退货，1 年内非人为损坏换新”等条款。

最后如何回答：把检索到的政策转成客服口吻，并明确下一步操作。

### 4. 代码执行类

用户任务：从一串 1000 个数字里找出质数并画分布图。

为什么不能直接答：这不是语言生成问题，而是批量计算和文件生成问题。

应该调用的工具：`python_interpreter`

观察：得到质数列表和图像文件。

最后如何回答：告诉用户筛出了多少个质数，并附上生成的图表或路径。

### 5. 实时信息类

用户任务：今天下午 3 点上海会不会下雨，要不要带伞。

为什么不能直接答：天气是实时数据。

应该调用的工具：`get_weather`

观察：返回降水概率、气温、天气状况。

最后如何回答：给出明确建议，例如“下午大概率有雨，建议带伞，并预留通勤时间”。

## 反例与边界

下面这些情况最容易误判：

1. “帮我写一个筛选质数的 Python 脚本”通常不需要执行工具，因为用户要的是代码，不是运行结果。
2. “请总结我刚发给你的这段文章”通常不需要检索工具，因为材料已经在上下文里。
3. “strawberry 里有几个 r”看似简单，但模型常在字符级任务上出错，更稳妥的是调用代码工具计数。
4. “今天是星期几”看似日常寒暄，但如果没有系统时间工具，模型可能会瞎猜。

## 常见错误

1. 工具名有了，但描述太模糊，模型不知道什么时候用、怎么传参。
2. 给太多工具，让模型在不相关选项里乱选。
3. 没有限制最大步数，导致无限循环。
4. 工具报错后不返回 observation，模型只能继续瞎编。
5. 模型自己伪造 observation，没有真正等待外部执行结果。
6. 输出格式不受约束，导致程序想要 JSON，模型却输出自然语言。
7. 长流程里忘了原始任务，后面越走越偏。
8. 给了高风险工具却没有审批点，例如发邮件、付款、删库。

> [!tip] 一句话改法
> 把工具描述写清楚，把工具数量压到最小，给出结构化输入输出，限制最大步数，并为高风险动作加人工确认。

## NotebookLM 式追问

### 概念澄清

1. 为什么“会说”不等于“会做”？
2. 工具调用解决的是知识问题、能力问题，还是权限问题？
3. ReAct 和普通 chain-of-thought 的差别在哪里？
4. 为什么 observation 是 agent 可靠性的关键环节？

### 迁移应用

1. 如果要让模型处理 Excel 文件，你需要给它哪些工具？
2. 如果用户要“先查天气，再改日程，再发提醒”，该如何拆成多步？
3. 一个搜索 agent 能不能被另一个主 agent 当成工具来调用？
4. 小模型和大模型在工具描述上要不要写得一样详细？

### 边界与风险

1. 哪些任务看起来适合调工具，其实直接回答更简单？
2. 哪些任务看起来能直接回答，但为了准确性仍应调工具？
3. 如果工具返回 5 万字 JSON，应该如何裁剪后再喂回模型？
4. 如果用户试图诱导模型调用危险工具，系统护栏应该放在哪一层？

## 作业

设计一个最小工具集，并说明每个工具的输入输出与权限边界。

建议把作业写成一个小设计稿，至少包含：

- 任务场景，例如“会议秘书”“研究助理”或“客服助手”
- 需要的最小工具集
- 每个工具的输入
- 每个工具的输出
- 哪些动作允许自动执行，哪些必须人工确认
- 工具失败时，模型应该如何汇报

## 课堂练习设计

任务：比较“购买 100 股 AAPL 股票”和“购买 1 盎司黄金”哪个更贵。

可用工具：

- `get_stock_price`
- `get_gold_price`
- `calculator`

评分点：

1. 是否先判断需要哪些外部信息。
2. 是否正确区分“查价格”和“做计算”是两个动作。
3. 是否在最终答案里说明比较依据，而不是只报一个结论。
