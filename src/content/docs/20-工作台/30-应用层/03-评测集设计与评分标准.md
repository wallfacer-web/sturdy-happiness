## 核心问题

怎样避免“我感觉这个 prompt 更好了”这种不可复查、不可迭代的判断？

## 这节学什么

学习先定义成功标准，再设计评测集、评分规则和回归测试流程。

## 为什么这节重要

很多提示词看起来“变好了”，只是因为它刚好在你眼前这一个例子上表现不错。只靠感觉调 prompt，会有三个常见问题：

- 你不知道它是不是普遍有效
- 你没法证明新版本比旧版本更好
- 你修好了一个问题，可能又弄坏了别的问题

只要提示词开始进入“可复用、可交付、可持续改进”的阶段，就必须建立评测思维。评测集和评分标准，本质上就是提示词工程里的最小质量控制系统。

## 关键概念

- eval set
- rubric
- pass/fail
- judge model
- regression test
- baseline

## 通俗解释

### 1. 什么是评测集

评测集，就是一组专门拿来测试 prompt 的样本。

它不是随便挑几条输入，而是有意识地覆盖不同情况：

- 常规样本
- 难样本
- 边界样本
- 容易出错的样本
- 故意扰动的样本

可以把它理解成 AI 的小型期末试卷。早期课程实验并不需要大规模数据，5 到 20 条高质量样本，通常就足够暴露大部分问题。

### 2. 什么是评分标准

评分标准，就是你用什么规则判断输出好不好。

最差的标准是：

- 我觉得不错
- 看起来还行

更好的标准是：

- 是否完成任务
- 是否事实正确
- 是否覆盖关键信息
- 是否符合输出格式
- 是否符合语气和角色要求
- 是否能直接使用

### 3. 什么是 pass/fail

有些任务不需要打 1 到 5 分，只要判断“过”还是“不过”。

典型场景：

- JSON 能不能被解析
- 输出里有没有必须字段
- 是否超过规定字数
- 是否含有禁止信息
- 是否按要求引用证据

这类任务适合做自动化检查。

### 4. 什么是 rubric

rubric 是更细的评分量表。它适合那些没有唯一标准答案、但仍然可以被拆成多个评估维度的任务。

例如评估一段课程介绍，可以从这些维度打分：

- 准确性
- 完整性
- 结构清晰度
- 语气适配度
- 可用性

每个维度可以设计 1 到 5 分，并且最好给出每档分数的定义，而不是只写一个名字。

### 5. 什么是 judge model

judge model 是让另一个模型来充当裁判，对输出做评分或比较。

但要注意，模型不会自动懂你的标准。你必须先把评分维度、评分档位、输出格式定义清楚，否则它只会给出空泛评价。

### 6. 什么是回归测试

回归测试指的是：你为了修一个 bug 改了 prompt，结果原来正常的场景又坏了。

这是提示词工程里最常见的坑之一。很多人只看当前那一条失败案例，修完就以为结束了，实际上系统整体已经退化。

## 不同任务，评测方式怎么选

### 1. 封闭式任务

这类任务通常存在比较明确的正确答案。

常见任务：

- 分类
- 抽取
- 结构化转换
- 格式校验

适合的评测方式：

- exact match
- pass/fail
- 字段覆盖率
- 精确率 / 召回率

示例：

任务是“从合同中提取付款日期”。如果答案应该是 `2026-04-15`，那模型输出错一天就是错，不需要靠感觉判断。

### 2. 开放式任务

这类任务没有唯一标准答案，更关注质量和适配性。

常见任务：

- 摘要
- 文案写作
- 教学解释
- 客服回复
- 复杂问答

适合的评测方式：

- rubric
- pairwise comparison
- human review
- judge model

示例：

任务是“写一段适合零基础学习者的 RAG 介绍”。这类输出不适合用 exact match，而是看它是否准确、易懂、结构清晰、没有过度术语化。

## 评测集怎么设计

一个能用的评测集，至少应该覆盖下面几类样本：

### 1. 常规样本

最典型、最常见的输入。它们代表你的主流使用场景。

### 2. 边界样本

不算完全异常，但容易让系统失稳。

例如：

- 很短的输入
- 很长的输入
- 信息不完整的输入
- 语气很冲的输入

### 3. 失败样本

过去已经出过错的输入。它们最适合拿来做回归测试。

### 4. 对抗样本

故意制造混淆或不规范输入，看系统能不能保持边界。

例如：

- 乱码
- 拼写错误
- 双重意图
- 不相关输入

### 5. 空答案样本

资料不足或任务本来就不该回答的情况。

这类样本特别适合 RAG、知识库问答和安全类任务，用来测试模型是否会老实承认“不知道”。

## 评分标准怎么写

### 1. 先写维度，不要先写总分

错误方式：

- 整体打个 8 分

更好的方式：

- 准确性 1-5
- 完整性 1-5
- 结构清晰度 1-5
- 风格适配度 1-5
- 可用性 1-5

### 2. 每一档都要有描述

不要只写 `1=差，5=好`。

更好的写法：

- 1 分：严重偏题，信息明显错误，不能使用
- 3 分：基本完成任务，但遗漏关键信息，需人工修改
- 5 分：准确完整，结构清晰，可直接交付

### 3. 尽量混合使用 pass/fail 和 rubric

很多任务不需要所有维度都打分。

例如在“结构化输出”任务里，可以这样组合：

- JSON 是否合法：pass/fail
- 必填字段是否齐全：pass/fail
- 字段内容是否准确：1-5 分
- 字段解释是否清楚：1-5 分

这比把所有东西都塞进一个总分更稳定。

## 10 个可直接教学的例子

### 例子 1：课程简介生成

输入 10 个课程主题，评估：

- 是否点明对象
- 是否说明学习目标
- 是否语气适合教学场景

### 例子 2：情感分类

准备：

- 明确正面
- 明确负面
- 混合情感
- 模糊废话

看模型能否稳定分类。

### 例子 3：JSON 提取

检查：

- 能否被解析
- 是否包含必填字段
- 字段类型是否正确

### 例子 4：合同信息抽取

检查：

- 金额、日期、甲乙方是否提对
- 原文没有写的内容是否乱编

### 例子 5：会议纪要摘要

检查：

- 是否保留关键决策
- 是否去掉口水话
- 是否新增原文没有的决策

### 例子 6：RAG 问答

专门放入“资料中没有答案”的问题，测试模型是否会明确说明证据不足。

### 例子 7：FAQ 生成

检查：

- 口径是否统一
- 是否过长
- 是否含有不允许的说法

### 例子 8：写作改写

比较不同 prompt 对语气、长度、正式程度的影响。

### 例子 9：工作流状态传递

测试中间结构化结果被修改后，下游节点是否还能正常消费。

### 例子 10：judge model 裁判

让 judge model 按 rubric 对两个版本的输出评分，再和人工判断对照，检查评分偏差。

## 反例

下面这些做法在课程里要明确指出来：

### 反例 1：只拿一条样本调 prompt

这最容易导致过拟合。你以为修好了，其实只是针对单例打补丁。

### 反例 2：所有任务都用一个总分

不同任务关注点不同。分类、摘要、JSON 输出、RAG 问答，根本不该共用同一种打分方式。

### 反例 3：让 judge model 自由发挥

如果你只说“请评价一下好不好”，它大概率会输出一段很圆滑但没法复用的空话。

### 反例 4：只做新功能测试，不做回归

这会导致旧问题不断重复出现。

### 方式二：用 gemma3:4b 跑一个最小评测循环

```python
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"

PROMPT_TEMPLATE = """
你是一个情感分类助手。
请把输入分类为：正面、负面、混合、无效。
只输出一个标签，不要解释。

输入：
{text}
""".strip()

eval_set = [
    {"id": 1, "input": "这个产品很好用，物流也很快。", "expected": "正面"},
    {"id": 2, "input": "客服回复太慢了，但最后问题解决了。", "expected": "混合"},
    {"id": 3, "input": "差评，再也不会买了。", "expected": "负面"},
    {"id": 4, "input": "啊？", "expected": "无效"},
]


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.2,
            "top_p": 0.9
        }
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


correct = 0
for row in eval_set:
    prompt = PROMPT_TEMPLATE.format(text=row["input"])
    pred = call_model(prompt)
    passed = pred == row["expected"]
    correct += int(passed)
    print("=" * 80)
    print("id:", row["id"])
    print("input:", row["input"])
    print("expected:", row["expected"])
    print("pred:", pred)
    print("PASS" if passed else "FAIL")

print("accuracy =", correct / len(eval_set))
```

### 方式八：批量比较两个 prompt 版本

```python
import csv
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"

eval_set = [
    {"id": 1, "input": "这个产品很好用，物流也很快。", "expected": "正面"},
    {"id": 2, "input": "客服回复太慢了，但最后问题解决了。", "expected": "混合"},

prompt_v1 = """
你是一个情感分类助手。
请把输入分类为：正面、负面、混合、无效。
只输出一个标签，不要解释。

输入：
{text}
""".strip()

prompt_v2 = """
你是一个严格的中文情感分类助手。
请把输入分类为：正面、负面、混合、无效。
规则：
1. 同时有正负信号时输出“混合”
2. 无明确语义或只有语气词时输出“无效”
3. 只输出一个标签，不要解释

输入：
{text}
""".strip()


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {"temperature": 0.2}
    }

    return response.json()["response"].strip()

for item in eval_set:
    pred_v1 = call_model(prompt_v1.format(text=item["input"]))
    pred_v2 = call_model(prompt_v2.format(text=item["input"]))
    rows.append({
        "id": item["id"],
        "input": item["input"],
        "expected": item["expected"],
        "pred_v1": pred_v1,
        "pred_v2": pred_v2,
        "pass_v1": pred_v1 == item["expected"],
        "pass_v2": pred_v2 == item["expected"]

with open("gemma3_4b_prompt_ab_compare.csv", "w", newline="", encoding="utf-8-sig") as f:

    )
    writer.writeheader()
    writer.writerows(rows)

print("saved: gemma3_4b_prompt_ab_compare.csv")
```

## 课堂实验方案

### 实验目标

让学习者亲手体验“改对一个样本，不等于整体变好”。

### 实验材料

- 一个基础 prompt
- 5 到 10 条评测样本
- 一个简单 rubric
- 一份回归记录表

### 实验步骤

1. 先写一个最小可用 prompt
2. 准备一组小型评测集
3. 先跑 baseline，记录失败点
4. 修改 prompt，只针对一个问题做改进
5. 重新跑完整评测集
6. 比较修改前后整体变化
7. 判断是否出现回归

### 建议观察点

- 准确率有没有真的提升
- 结构化输出是否更稳定
- 新版 prompt 是否牺牲了其他样本
- 哪些样本最容易被忽视

## 课堂流程

1. 先让学生说出他们平时如何判断 prompt 好不好
2. 引出“感觉不错”为什么不可靠
3. 带学生把一个任务拆成评测维度
4. 现场建立 5 到 10 条小评测集
5. 跑 baseline
6. 修改 prompt
7. 做回归测试
8. 复盘哪些改动是真的有效，哪些改动只是针对单例优化

## 教师提醒

### 提醒 1

不要只用当前失败的那一条样本调 prompt，这会导致过拟合。

### 提醒 2

judge model 不是万能裁判。没有 rubric，它只会给出看起来很完整、实际上没法复用的评价。

### 提醒 3

测试集不是越大越好。教学初期先做 5 到 20 条高质量样本，比盲目堆数量更有价值。

### 提醒 4

对写作、摘要、教学解释这类开放式任务，人工评审仍然是必要的，不能完全交给自动评分。

### 提醒 5

评测一定要保留 baseline。没有基线，后面的“优化”就失去了比较对象。

## 常见误区

- 误以为“看起来不错”就等于 prompt 好
- 误以为修复当前问题就等于系统整体变好
- 误以为所有任务都适合用一个总分
- 误以为自动化评分可以完全替代人工判断
- 误以为测试集越大越专业，忽略样本多样性

## 课后延伸

- 给你最常用的一条 prompt 建一个最小评测集
- 区分哪些任务适合 pass/fail，哪些任务适合 rubric
- 试着为一个开放式任务写一个 judge model 评分提示词
- 结合 [提示词评测模板](../../40-模板/提示词评测模板/) 整理自己的评测工作台

## 作业

提交一个最小评测集、一个评分 rubric，以及一次版本修改前后的对比记录。
