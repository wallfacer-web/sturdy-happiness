## 核心问题

复杂任务为什么要拆，怎样拆才不会增加无效步骤。

## 这节学什么

学习只保留必要步骤，让任务链既清晰又不臃肿。

## 为什么这节重要

很多人一开始用大模型处理复杂任务时，会本能地把所有要求塞进一个超长 prompt 里，希望模型“一次性全做完”。结果往往是：

- 每个部分都提到了，但都不够深
- 格式混乱
- 中间逻辑跳步
- 后面的要求覆盖前面的要求

任务拆解和提示词链，就是把复杂任务从“一团命令”变成“一个流程”。

## 关键概念

- decomposition
- sequential prompts
- intermediate outputs

## 通俗解释

### 1. 什么是任务拆解

任务拆解就是把一个复杂目标拆成若干个更小、更清楚的步骤。

例如，“做一份课程方案”其实不是一个动作，而是至少包含：

- 理解资料
- 明确目标
- 设计结构
- 生成成稿

### 2. 什么是提示词链

提示词链就是把这些步骤串起来，让上一步的输出成为下一步的输入。

最简单的链式结构就是：

```text
步骤1输出 -> 步骤2输入
步骤2输出 -> 步骤3输入
步骤3输出 -> 最终结果
```

### 3. 为什么复杂任务不适合一轮做完

因为复杂任务往往同时包含：

- 信息提取
- 判断分析
- 内容生成
- 格式控制

这些任务混在一起，会让模型负担过重。拆开之后，每一步的目标更清晰，模型更容易做好。

## 五个核心概念

### 1. 步骤拆解

先找出完成目标必须经历的关键步骤。

### 2. 中间结果

每一步的输出都不是废话，而是下一步的原材料。

### 3. 状态传递

要确保上一步的重要结果被准确传到下一步，而不是依赖模型“自己记住”。

### 4. 串行流程

很多提示词链是线性的：A -> B -> C。

### 5. 回退点

如果链条某一步做偏了，不应该从头重来，而应该回到出错节点修正。

## 什么任务适合拆链

### 适合拆链的任务

- 长文总结
- 多文档分析
- 研究设计
- 复杂写作
- 项目规划
- 报告生成
- FAQ整理
- 知识卡片生成

### 不一定要拆链的任务

- 简单翻译
- 单轮提取
- 简单分类
- 单一格式转换

判断标准很简单：如果你在一个 prompt 里已经写出“先……再……最后……”很多次，通常就该拆链了。

## 典型练习

把“做一份课程方案”拆成资料提炼、目标定义、模块设计、成稿四步。

## 最小示例

先让模型提炼资料，再要求它基于提炼结果生成课程结构，而不是一步到位。

### 方式三：Python 最小提示词链

```python
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.3,
            "top_p": 0.9
        }
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


step1_prompt = """
请基于下面资料，提炼目标学习者、课程目标和使用场景。

资料：
这是一门面向零基础知识工作者的提示词工程课程，强调提问设计、上下文组织、评测与工作流。
"""

step1_result = call_model(step1_prompt)
print("===== 第一步结果 =====")
print(step1_result)

step2_prompt = f"""
基于下面中间结果，生成课程模块结构。

中间结果：
{step1_result}
"""

step2_result = call_model(step2_prompt)
print("\n===== 第二步结果 =====")
print(step2_result)
```

### 方式四：完整四步提示词链

```python
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.3,
            "top_p": 0.9
        }
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


material = """
这是一门面向零基础知识工作者的提示词工程课程，
强调提问设计、上下文组织、评测与工作流。
"""

step1 = call_model(f"""
请提炼以下资料中的目标学习者、能力目标和应用场景：
{material}
""")

step2 = call_model(f"""
基于以下结果，设计课程模块：
{step1}
""")

step3 = call_model(f"""
基于以下课程模块，设计每个模块的典型练习：
{step2}
""")

step4 = call_model(f"""
基于以下内容，生成一份结构化课程简介：
课程模块：
{step2}

典型练习：
{step3}
""")

print("===== step1 =====")
print(step1)
print("\n===== step2 =====")
print(step2)
print("\n===== step3 =====")
print(step3)
print("\n===== step4 =====")
print(step4)
```

## 课堂实验方案

### 实验目标

让学生直观看到“一个超长提示词全做完”和“拆成三步提示词链”之间的差异。

### 实验任务

分析 50 条产品负面评价，并给出改进建议。

### 对照组

直接提问：

```text
请分析这些客户评价，找出主要问题并给出详细解决建议。
```

### 实验组

拆成三步：

1. 提取主要负面主题
2. 分析根本原因
3. 生成改进建议

### 观察点

- 输出是否更具体
- 逻辑是否更清楚
- 是否减少“正确的废话”
- 是否更容易修改和回退

## 课堂流程

1. 找出任务中的隐含子任务
2. 画出步骤链
3. 为每一步定义输入输出

## 教师提醒

### 提醒1

如果一个 prompt 里已经出现很多“先、再、然后、最后”，通常就说明该拆链了。

### 提醒2

中间结果要检查，不要盲目直接送入下一步，否则错误会被放大。

### 提醒3

链条越长，越需要显式传递状态，而不要默认模型能一直记住。

### 提醒4

不是所有任务都值得拆。简单任务直接结构化提示更高效。

## 常见误区

- 误以为拆得越细越好
- 误以为链条一旦写出就不用检查中间结果
- 误以为模型会自动记住前面所有约束
- 误以为提示词链只能程序化实现
- 误以为复杂任务一定能靠一个巨型 prompt 解决
