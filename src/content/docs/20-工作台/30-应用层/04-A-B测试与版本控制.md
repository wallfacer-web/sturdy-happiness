## 核心问题

怎样让提示词优化有版本、有证据、有回滚路径，而不是一遍遍改完就忘？

## 这节学什么

学习如何设计 prompt 的 A/B 对照实验，如何记录 baseline、变量、评测结果和结论，以及如何建立一个最小可用的版本控制习惯。

## 为什么这节重要

很多人做提示词优化时，其实没有真正做实验，只是在不断“改一改再试试”。这种做法有四个问题：

- 你不知道到底是哪一个改动起了作用
- 你没法证明 v2 比 v1 更好
- 你无法区分“整体提升”和“偶然命中”
- 一旦新版本变差，你也说不清该回退到哪里

只要进入持续迭代阶段，prompt 就不是一句临时写的话，而是一个需要管理的工作对象。A/B 测试负责回答“哪个版本更好”，版本控制负责回答“它为什么变好、什么时候变坏、该回退到哪一版”。

## 关键概念

- A/B test
- baseline
- variable
- control
- versioning
- prompt logbook
- rollback
- regression

## 通俗解释

### 1. 什么是 A/B 测试

A/B 测试就是把两个版本放在同样条件下比较。

在提示词工程里，最简单的形式是：

- 保留旧版本 `v1`
- 写出新版本 `v2`
- 用同一批样本去跑
- 用同一套指标去比

如果没有“同一批样本”和“同一套标准”，那就不叫对照实验，只能算随手试一试。

### 2. 什么是 baseline

baseline 就是当前可用的基线版本。

它不一定完美，但它是后续所有优化的比较对象。没有 baseline，你就无法判断“优化”到底有没有真的发生。

### 3. 什么是变量控制

做 A/B 测试时，最关键的一点是一次只改一个主要变量。

例如：

- 只改角色设定
- 只改输出格式
- 只改 few-shot 示例
- 只改约束条件

如果你同时改了角色、格式、示例和语气，那最后即使结果更好，你也不知道到底是哪一部分在起作用。

### 4. 什么是版本控制

版本控制不只是 Git。对课程学习者来说，更重要的是养成“每次迭代都留下记录”的习惯。

最小版本记录至少要回答这些问题：

- 这一版叫什名
- 在上一版基础上改了什么
- 为什么改
- 用什么样本测
- 结果怎么样
- 下一步准备怎么改

### 5. 什么是回滚

回滚不是失败，而是工程常态。

如果 v2 在某个样本上变好了，但在整体上更差，那就应该明确回到 v1，或者从 v1.1 重新分支，而不是硬把坏版本留下来。

## A/B 测试怎么设计

### 1. 先定义实验目标

不要一上来就说“我想让它更好”，这太模糊。

更好的说法是：

- 我想让摘要更短，但不要漏掉关键决策
- 我想让分类更稳定，特别是混合情感样本
- 我想让 JSON 输出解析成功率更高
- 我想让课程介绍更适合零基础学习者

### 2. 明确 A 和 B 的差异

写清楚两个版本到底差在哪。

例如：

- A：只有任务描述
- B：加入角色 + 输出格式 + 字段约束

或者：

- A：zero-shot
- B：few-shot

或者：

- A：无兜底规则
- B：加入“资料不足时明确说不知道”

### 3. 固定测试样本

不要 A 跑一批样本，B 跑另一批样本。这样结论没有可比性。

### 4. 固定指标

不要 A 看准确率，B 看主观感觉。两边要使用同一套评估方式。

### 5. 记录结论，不只记录结果

实验结束后，不要只写“B 更好”。要写清楚：

- B 好在哪
- 哪些样本没变
- 哪些样本退化
- 下一轮应该继续改哪里

## 版本控制怎么做

### 1. 建议使用清晰的命名

例如：

- `v1-baseline`
- `v1.1-add-json-format`
- `v1.2-add-fallback-rule`
- `v2-fewshot-classification`

这比“新版”“最新版”“最终版2”要清楚得多。

### 2. 每次改动写一句变更说明

例如：

- 加入固定输出字段，提升 JSON 稳定性
- 增加混合情感示例，减少误判
- 删除过强限制，避免把有效输入误判为无效

### 3. 建一个 Prompt Logbook

建议记录这些字段：

- version
- date
- task
- change
- hypothesis
- eval_set
- metric
- result
- regression
- next_step

### 4. 保留失败版本

失败版本也有价值，因为它们能说明哪些思路无效。对课程教学来说，这反而是非常好的案例。

## 10 个可直接教学的例子

### 例子 1：课程介绍生成

- A：只说“写课程简介”
- B：增加目标学习者、语气、长度和结构要求
- 比较：哪一版更适合教学宣传页

### 例子 2：情感分类

- A：zero-shot
- B：few-shot
- 比较：混合情感样本是否更稳定

### 例子 3：摘要任务

- A：要求简短
- B：要求“保留关键决策，不要口水话”
- 比较：是否减少信息丢失

### 例子 4：JSON 输出

- A：要求输出 JSON
- B：要求固定字段并禁止解释文字
- 比较：解析成功率

### 例子 5：RAG 问答

- A：直接回答
- B：要求必须依据证据并允许回答“不知道”
- 比较：幻觉率

### 例子 6：客服回复

- A：普通礼貌语气
- B：加入共情和兜底规则
- 比较：情绪化输入下的稳定性

### 例子 7：写作改写

- A：只要求“更正式”
- B：明确目标读者、文风和长度
- 比较：语气一致性

### 例子 8：任务拆解

- A：单步完成
- B：两步链式生成
- 比较：复杂任务中的完整性

### 例子 9：工作流状态输出

- A：自然语言中间结果
- B：JSON 中间结果
- 比较：下游节点消费稳定性

### 例子 10：Judge 模型评分

- A：自由评价
- B：按 rubric 打分
- 比较：评分一致性

## 反例

### 反例 1：同时改一堆东西

这会让实验失去解释力。结果变好也不知道为什么，结果变坏也不知道错在哪。

### 反例 2：A 和 B 用不同测试集

这不是对照实验，只是两个独立试玩。

### 反例 3：只记“结论”，不记过程

过几天回头看时，你根本不知道当时为什么做那个判断。

### 反例 4：只看平均分，不看失败样本

平均分提升不代表关键边界场景没退化。

### 方式一：最小 A/B 对照实验

```python
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"

eval_set = [
    {"id": 1, "input": "这个产品很好用，物流也很快。", "expected": "正面"},
    {"id": 2, "input": "客服回复太慢了，但最后问题解决了。", "expected": "混合"},
    {"id": 3, "input": "差评，再也不会买了。", "expected": "负面"},
    {"id": 4, "input": "啊？", "expected": "无效"}
]

prompt_a = """
你是一个情感分类助手。
请把输入分类为：正面、负面、混合、无效。
只输出一个标签，不要解释。

输入：
{text}
""".strip()

prompt_b = """
你是一个严格的中文情感分类助手。
请把输入分类为：正面、负面、混合、无效。
规则：
1. 同时出现正负信号时输出“混合”
2. 语义不完整、只有语气词或无法判断时输出“无效”
3. 只输出一个标签，不要解释

输入：
{text}
""".strip()


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.2,
            "top_p": 0.9
        }
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


for item in eval_set:
    pred_a = call_model(prompt_a.format(text=item["input"]))
    pred_b = call_model(prompt_b.format(text=item["input"]))
    print("=" * 80)
    print("id:", item["id"])
    print("input:", item["input"])
    print("expected:", item["expected"])
    print("pred_a:", pred_a)
    print("pred_b:", pred_b)
```

```python
]

prompt_a = """
你是一个情感分类助手。
请把输入分类为：正面、负面、混合、无效。
只输出一个标签。

输入：
{text}
""".strip()

prompt_b = """
你是一个严格的中文情感分类助手。
请把输入分类为：正面、负面、混合、无效。
规则：
1. 同时出现正负信号时输出“混合”
2. 语义不完整、只有语气词或无法判断时输出“无效”
3. 只输出一个标签，不要解释

输入：
{text}
""".strip()


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {"temperature": 0.2}
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


score_a = 0
score_b = 0

for item in eval_set:
    pred_a = call_model(prompt_a.format(text=item["input"]))
    pred_b = call_model(prompt_b.format(text=item["input"]))
    score_a += int(pred_a == item["expected"])
    score_b += int(pred_b == item["expected"])

print("score_a =", score_a)
print("score_b =", score_b)
print("winner =", "A" if score_a > score_b else "B" if score_b > score_a else "tie")
```

### 方式三：保存 A/B 结果到 CSV

```python
import csv
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"

eval_set = [
    {"id": 1, "input": "这个产品很好用，物流也很快。", "expected": "正面"},
    {"id": 2, "input": "客服回复太慢了，但最后问题解决了。", "expected": "混合"},
    {"id": 3, "input": "差评，再也不会买了。", "expected": "负面"},
    {"id": 4, "input": "啊？", "expected": "无效"}
]

prompt_a = "你是一个情感分类助手。请把输入分类为：正面、负面、混合、无效。只输出一个标签。\n\n输入：\n{text}"
prompt_b = "你是一个严格的中文情感分类助手。请把输入分类为：正面、负面、混合、无效。规则：1. 同时出现正负信号时输出“混合” 2. 语义不完整、只有语气词或无法判断时输出“无效” 3. 只输出一个标签，不要解释。\n\n输入：\n{text}"


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {"temperature": 0.2}
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


rows = []
for item in eval_set:
    pred_a = call_model(prompt_a.format(text=item["input"]))
    pred_b = call_model(prompt_b.format(text=item["input"]))
    rows.append({
        "id": item["id"],
        "input": item["input"],
        "expected": item["expected"],
        "pred_a": pred_a,
        "pred_b": pred_b,
        "pass_a": pred_a == item["expected"],
        "pass_b": pred_b == item["expected"]
    })

with open("gemma3_4b_ab_test_results.csv", "w", newline="", encoding="utf-8-sig") as f:
    writer = csv.DictWriter(
        f,
        fieldnames=["id", "input", "expected", "pred_a", "pred_b", "pass_a", "pass_b"]
    )
    writer.writeheader()
    writer.writerows(rows)

print("saved: gemma3_4b_ab_test_results.csv")
```

## 课堂实验方案

### 实验目标

让学习者真正经历一次“有控制变量的 prompt 对照实验”。

### 实验材料

- 一个 baseline prompt
- 一个改进版 prompt
- 5 到 10 条固定评测样本
- 一张实验记录表

### 实验步骤

1. 先确定一个明确目标
2. 写出 baseline 版本
3. 只改一个主要变量，写出新版本
4. 用同一评测集跑 A/B
5. 记录结果，不只看平均分，也看失败样本
6. 判断是否升级版本、继续优化还是回滚

### 观察点

- 新版本到底改善了什么
- 它是否造成新的回归
- 哪类样本最敏感
- 哪些改动是高收益，哪些只是装饰

## 课堂流程

1. 先展示一个“看起来更长更复杂，但不一定更好”的 prompt
2. 引出为什么要做 A/B 测试
3. 带学生定义 baseline 和变量
4. 用固定样本跑两版
5. 记录结果和结论
6. 讨论是否应该升级版本或回滚

## 教师提醒

### 提醒 1

一次只改一个主要变量。否则这节课会变成“感觉学”，而不是实验课。

### 提醒 2

不要因为 B 版本写得更长、更复杂，就默认它更先进。

### 提醒 3

版本控制不要求一开始就上复杂工具，先把 logbook 记清楚，比什么都重要。

### 提醒 4

即使 B 赢了，也要看它赢在哪些样本上，输在哪些样本上。

### 提醒 5

回滚不是丢脸，而是提示词工程的正常动作。

## 常见误区

- 误以为“新版”天然比“旧版”好
- 误以为 prompt 变长就等于 prompt 变强
- 误以为只要结果偶尔更好，就算 A/B 成功
- 误以为不需要记录实验过程
- 误以为回滚意味着前面的实验白做了
