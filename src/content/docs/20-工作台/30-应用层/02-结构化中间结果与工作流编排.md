## 核心问题

怎样让上一步输出稳定地喂给下一步，而不是每次重新解释。

## 这节学什么

学习让中间结果可解析、可复用、可检查。

## 为什么这节重要

很多人在做提示词链时，第一步已经拆出来了，第二步也设计了，但真正跑起来还是不稳定。最常见的原因不是模型不够强，而是中间结果太“像人话”，不够“像状态”。

比如上一步输出：

```text
好的，我已经为你整理出以下重点……
```

这种自然语言对人类友好，但对下一步工作流很不友好。结构化中间结果的核心就是：让每一步的输出都像一个可传递、可检查、可消费的状态对象，而不是一段临时聊天。

## 关键概念

- schema
- state passing
- workflow orchestration

## 通俗解释

### 1. 为什么中间结果必须结构化

因为自然语言有三个问题：

- 容易歧义
- 格式不稳定
- 难以被程序和后续步骤稳定读取

而 JSON、表格、字段化输出有三个优势：

- 字段固定
- 可直接解析
- 更容易验证

### 2. 什么是 schema

schema 就是一张“答题卡模板”。它提前规定：

- 这一步必须输出哪些字段
- 每个字段代表什么
- 哪些字段必填

例如：

```json
{
  "audience": "",
  "goals": [],
  "constraints": [],
  "materials": []
}
```

### 3. 什么是 state

state 就是当前工作流已经知道的有效状态。

你可以把它理解为“流程记事本”。工作流不是靠模型回忆前面发生了什么，而是每一步显式携带状态进入下一步。

### 4. 什么是 handoff

handoff 就是交接。上一节点产出的结构化结果，被提取并交给下一节点继续处理。

### 5. 什么是 validation

validation 就是质检。比如：

- 它是不是合法 JSON
- 有没有漏字段
- 字段类型对不对

如果不验证，错误会直接沿着链条往后放大。

### 6. 什么是 fallback

fallback 就是兜底方案。如果这一步格式错了、字段缺了、状态不完整，就不要硬往下跑，而是：

- 让模型重试
- 回退到上一步修正
- 或交给人工处理

## 结构化为什么比自然语言中间结果更适合工作流

因为结构化输出的目标不是“好读”，而是“可执行”。

例如：

### 不稳定的自然语言

```text
这个课程大概适合零基础学生，重点大概有四个，可能还需要练习和案例。
```

### 稳定的结构化状态

```json
{
  "audience": "零基础知识工作者",
  "module_count": 4,
  "needs_examples": true,
  "needs_practice": true
}
```

第二种格式更容易被后续步骤直接消费。

## 五个核心概念展开

### 1. Schema

规定输出长什么样。

### 2. State Passing

让前一步的结果不是“留在聊天历史里”，而是明确变成下一步的输入。

### 3. Workflow Orchestration

把多个节点按规则组织成一个流程，而不是临时拼接。

### 4. Validation

防止不合法结果流入后面步骤。

### 5. Fallback

防止单个节点出错后整条链条直接崩掉。

### 方式三：Python 最小状态传递

```python
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.2,
            "top_p": 0.9
        }
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


step1 = call_model("""
请输出 JSON，字段为 audience、goals、constraints。
资料：这是一门面向零基础知识工作者的提示词工程课程。
不要输出解释文字。
""")

print("===== step1 =====")
print(step1)

step2 = call_model(f"""
基于以下 JSON 状态，生成课程模块。
状态：
{step1}
""")

print("\n===== step2 =====")
print(step2)
```

### 方式五：Python fallback 示例

```python
import json
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"


def call_model(prompt):
    payload = {
        "model": MODEL,
        "prompt": prompt,
        "stream": False,
        "options": {
            "temperature": 0.2,
            "top_p": 0.9
        }
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


prompt = """
请输出 JSON，字段为 task_description、assigned_to、deadline。
不要输出任何解释。
文本：张三负责周五前完成首页初稿。
"""

result = call_model(prompt)

try:
    parsed = json.loads(result)
    print("解析成功：", parsed)
except json.JSONDecodeError:
    retry_prompt = prompt + "\n注意：上一次输出不是合法 JSON，请严格只输出 JSON。"
    retry_result = call_model(retry_prompt)
    print("重试结果：", retry_result)
```

## LangChain 与 LangGraph 示例

下面这部分使用的是 LangChain / LangGraph 当前官方主线思路。这里优先演示两类能力：

- 结构化输出
- 状态图工作流

参考文档：

- https://docs.langchain.com/oss/python/langchain/structured-output
- https://reference.langchain.com/python/langchain-core/output_parsers
- https://reference.langchain.com/python/langgraph/graph/state/StateGraph

### 1. LangChain 结构化输出示例

这个例子展示怎样让模型直接返回一个结构化对象，而不是一段自然语言。

```python
from pydantic import BaseModel, Field
from langchain.chat_models import init_chat_model


class MeetingTask(BaseModel):
    task_description: str = Field(description="具体任务")
    assigned_to: str = Field(description="负责人")
    deadline: str | None = Field(description="截止日期")


model = init_chat_model(
    "gpt-4o-mini",  # 这里换成你实际可用的模型
    model_provider="openai",
)

structured_model = model.with_structured_output(MeetingTask)

result = structured_model.invoke(
    "从这句话中提取任务信息：张三负责在周五前完成首页初稿。"
)

print(result)
```

这类写法的价值是：你拿到的是字段对象，不必自己再解析一段自然语言。

### 2. LangChain 简单 Runnable 链

这个例子展示如何把两个步骤串起来。

```python
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain.chat_models import init_chat_model

model = init_chat_model(
    "gpt-4o-mini",
    model_provider="openai",
)

extract_prompt = ChatPromptTemplate.from_template(
    "请提炼下面资料中的目标学习者和能力目标：{text}"
)

design_prompt = ChatPromptTemplate.from_template(
    "基于下面中间结果，生成课程模块：{state}"
)

parser = StrOutputParser()

extract_chain = extract_prompt | model | parser

state = extract_chain.invoke(
    {"text": "这是一门面向零基础知识工作者的提示词工程课程。"}
)

design_chain = design_prompt | model | parser

result = design_chain.invoke({"state": state})
print(result)
```

### 3. LangGraph StateGraph 最小工作流

这个例子更接近“工作流编排”。

```python
from typing_extensions import TypedDict
from langgraph.graph import StateGraph, START, END


class WorkflowState(TypedDict):
    source_text: str
    extracted_state: str
    course_modules: str


def extract_node(state: WorkflowState):
    text = state["source_text"]
    return {
        "extracted_state": f"从资料中提炼出的状态: {text}"
    }


def design_node(state: WorkflowState):
    extracted = state["extracted_state"]
    return {
        "course_modules": f"基于 {extracted} 生成课程模块"
    }


graph = StateGraph(WorkflowState)
graph.add_node("extract", extract_node)
graph.add_node("design", design_node)

graph.add_edge(START, "extract")
graph.add_edge("extract", "design")
graph.add_edge("design", END)

app = graph.compile()

result = app.invoke(
    {"source_text": "这是一门面向零基础知识工作者的提示词工程课程。"}
)

print(result)
```

这个例子里最关键的不是模型，而是状态的显式流动：

- `source_text`
- `extracted_state`
- `course_modules`

### 4. LangChain / LangGraph 在本节里最值得学什么

如果你是学习者，先抓住两件事：

1. 用结构化输出替代自然语言中间结果
2. 用显式状态图替代“我希望模型记住前面说过什么”

## 课堂实验方案

### 实验目标

让学生直观看到“自然语言中间结果”和“结构化中间结果”在后续流程中的稳定性差异。

### 实验任务

把一段会议纪要转成任务清单，再基于任务清单生成通知邮件。

### 对照组

第一步输出自然语言总结，再直接喂给第二步。

### 实验组

第一步只输出 JSON：

- `task_description`
- `assigned_to`
- `deadline`

再让第二步读取 JSON 生成邮件。

### 观察点

- 第二步是否更容易读取
- 是否更少丢任务
- 是否更少出现废话
- 是否更适合程序接入

## 课堂流程

1. 先看非结构化中间结果的问题
2. 再改成固定 schema
3. 讨论什么字段必须保留

## 教师提醒

### 提醒1

不要只说“请结构化输出”，最好直接给 schema。

### 提醒2

如果工作流有后续步骤，一定要做 validation。

### 提醒3

状态不是越多越好，应该只保留后续步骤真正需要的字段。

### 提醒4

结构化工作流不是为了炫技，而是为了减少歧义和返工。

## 常见误区

- 误以为自然语言中间结果也足够稳定
- 误以为只要是 JSON 就一定可靠
- 误以为不需要做格式验证
- 误以为状态应该把所有上下文都传下去
- 误以为工作流编排就是把 prompt 按顺序排一下
