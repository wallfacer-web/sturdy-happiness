## 核心问题

怎样通过角色定义、边界约束和输出格式提高响应一致性。

## 这节学什么

学习什么时候该给角色，什么时候该给约束，什么时候该直接规定输出格式。

## 为什么这节重要

很多初学者已经知道“要把任务写清楚”，但写清楚之后仍然常见两个问题：

1. 模型虽然完成了任务，但口吻不对、深度不对
2. 模型虽然讲了很多，但结果不方便直接使用

这时候就要进入更细一层的控制：角色、约束和输出格式。它们不是装饰，而是让模型从“会回答”进一步变成“回答得像你要的那样，并且结果可落地”的关键手段。

## 关键概念

- role prompt

## 通俗解释

### 1. 角色提示是什么

角色提示就是告诉模型“你现在以什么身份、什么视角来完成任务”。例如：

- 你是一名高中物理老师
- 你是一名严苛的匿名审稿人
- 你是一名产品运营分析师

角色的真正作用，不是让模型“演戏”，而是帮助它调用更适合当前任务的语言风格、专业标准和判断框架。

### 2. 什么时候角色有用

角色有用的场景：

- 你需要模型采用特定专业视角
- 你需要模型控制语气或风格
- 你需要模型按某种评价标准来判断内容

例如，“帮我评审这个研究设想”和“请以匿名审稿人的标准批判性评审这个研究设想”，输出深度通常不同。

### 3. 什么时候角色只是装饰

如果任务本身是高度确定性的，例如：

- 提取时间
- 分类标签
- 转成 JSON
- 把参数整理成表格

这类任务里写“你是一位顶级专家”常常没有实际帮助。因为任务重点不在视角，而在准确执行。这里更关键的是约束和输出格式。

### 4. 约束是什么

约束就是给模型设边界。它回答的是：

- 什么能做
- 什么不能做
- 结果不能超出哪些范围

例如：

- 只能基于输入材料
- 不要编造
- 不要使用术语
- 不超过 200 字
- 只能输出标签，不要解释

约束的本质是减少跑偏。

### 5. 输出格式为什么重要

输出格式决定结果能不能直接用。如果你只说“帮我分析一下”，模型往往输出一大段自然语言；但如果你明确要求：

- Markdown 表格
- JSON
- 三点列表
- 分级标题

结果就更容易直接贴进文档、表格、程序或后续工作流。

### 6. 为什么这三个要素要一起看

一个好提示词通常不是只靠其中一个要素，而是组合使用：

- 用角色控制“站在什么角度说”
- 用约束控制“哪些内容不能乱来”
- 用格式控制“结果怎么交付”

## 典型练习

要求模型同时输出表格版、JSON 版和教学版解释。

## 最小示例

“你是一名课程助教，请把下面概念解释成适合零基础学生的版本，并输出 JSON：定义、例子、误区。”

## 8个可直接上手的例子

### 例子1：教学解释

```text
你是一名高中信息技术老师。
请解释什么是向量数据库。
要求：面向零基础学生，必须给生活化比喻。
输出格式：按“定义、比喻、最小例子、常见误区”四部分输出。
```

这里角色有用，因为它决定了语言难度和讲解方式。

### 例子2：研究辅助

```text
你是一名严苛的匿名学术审稿人。
请评审以下研究设想。
要求：只基于我提供的文本，不要补充外部事实。
输出格式：按“研究问题、方法缺口、潜在偏差、修改建议”四部分输出。
```

这里角色有用，因为它决定模型采用“批判性评价”的标准。

### 例子3：信息提取

```text
请从下面的合同文本中提取核心字段。
要求：如果原文未提及，就写“原文未提及”。
输出格式：字段包括出租方、承租方、租金、付款方式、违约条款。
```

这里角色不重要，真正关键的是约束和固定字段。

### 例子4：Markdown 表格输出

```text
请比较两款手机的核心参数。
要求：只保留客观参数，不要加入评价。
输出格式：Markdown表格，列为“维度、手机A、手机B、差异点”。
```

这里输出格式直接决定结果能否一键贴到文档中。

### 例子5：JSON 输出

```text
请从下面文本中提取联系人信息。
要求：不要输出解释，不要输出代码块标记。
输出格式：JSON，键名固定为 name、title、company、email。
```

这里约束和格式都比角色重要。

### 例子6：写作改写

```text
你是一名商业写作编辑。
请把下面口语化项目汇报改写成正式备忘录。
要求：保留原文事实，不要新增数据。
输出格式：背景、关键发现、下一步行动。
```

这里角色有用，因为它影响语气和组织方式。

### 例子7：分类任务

```text
请对以下客户评论进行分类。
要求：只能输出“正面、负面、中立”之一，不要解释。
输出格式：每条评论输出一个标签。
```

这类任务里角色通常是次要的，约束最关键。

### 例子8：计划生成

```text
你是一名企业培训规划师。
请为新员工设计一门时间管理培训课。
要求：课程具备互动性，总时长控制在2小时。
输出格式：按“课程目标、模块安排、互动活动、时间分配”输出。
```

这里三个要素都有效：角色给专业视角，约束给现实边界，格式让结果可执行。

### 方式四：比较“只有任务”和“加角色+约束+格式”

这段代码很适合课堂演示，因为它会让学生直接看到差异。

```python
import requests

URL = "http://localhost:11434/api/generate"
MODEL = "gemma3:4b"

prompts = {
    "版本A_只有任务": "解释什么是向量数据库。",
    "版本B_加角色约束格式": """
你是一名课程助教。
请解释什么是向量数据库。
要求：
1. 面向零基础学生
2. 必须给生活化比喻
3. 不要使用术语堆砌
输出格式：
- 一句话定义
- 生活比喻
- 最小例子
- 常见误区
"""
}


def run_prompt(prompt_text):
    payload = {
        "model": MODEL,
        "prompt": prompt_text,
        "stream": False,
        "options": {
            "temperature": 0.4,
            "top_p": 0.9
        }
    }
    response = requests.post(URL, json=payload, timeout=120)
    response.raise_for_status()
    return response.json()["response"].strip()


for name, prompt_text in prompts.items():
    print("=" * 80)
    print(name)
    print("=" * 80)
    print(run_prompt(prompt_text))
    print()
```

### 方式六：批量比较不同输出格式

## 课堂练习方案

### 练习目标

让学生观察：同一个任务，加入角色、约束和输出格式后，结果会怎样变化。

### 练习任务

先让学生输入这条基础提示词：

```text
帮我写一份关于电动汽车市场的报告。
```

然后要求学生把它重写成包含以下部分的版本：

- 角色
- 约束
- 输出格式

### 练习步骤

1. 先运行模糊版本
2. 记录问题：是否空泛、是否冗长、是否难直接使用
3. 重写为结构化版本
4. 再运行一次
5. 比较前后差异

### 观察点

- 是否更有针对性
- 是否更容易直接复制到报告
- 是否更少空话
- 是否更符合目标受众

## 课堂流程

1. 比较“只有角色”和“角色 + 约束 + 格式”的差异
2. 讨论哪些角色设定是无效装饰
3. 演示结构化输出的好处

## 教师提醒

### 提醒1

不要把“你是专家”当成万能咒语。角色越具体越有用，越空泛越没意义。

### 提醒2

约束最好既有正面要求，也有负面边界。不要只写“不要这样”，也要写“应该怎样”。

### 提醒3

如果你特别在意输出格式，最好直接给格式骨架，甚至给一个微型示例。

### 提醒4

当任务本身是提取、分类、转格式时，优先保证约束和格式清晰，不要把精力浪费在花哨角色上。

## 常见误区

- 误以为角色越夸张越有效
- 误以为所有任务都需要角色设定
- 误以为只写“不要编造”就足够
- 误以为模型会自动按你心中的表格或 JSON 输出
- 误以为格式要求只是排版问题，和质量无关

## 课后延伸

- 试着给同一个任务设计“教师版、分析师版、编辑版”三种角色提示，比较差异
- 固定任务和角色，只替换输出格式，比较普通段落、表格、JSON 的可用性
- 结合 [结构化提示词](../../../30-知识卡片/10-概念卡/结构化提示词/) 整理你常用的格式模板
